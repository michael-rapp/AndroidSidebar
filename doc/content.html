1 Implementation details

The sidebar, which is provided by this library, is implemented as a custom view within the class Sidebar. This class is delegated from the class ViewGroup of the Android API. Just like its parent class, the subclass provides three different constructors. The one of them, which only expects a Context object to be passed as a parameter, can be used to instantiate the Sidebar programatically, the others are implicitly used by the Android API, when the view is defined by using a XML resource file.

Furthermore, the class Sidebar overrides the method onSaveInstanceState():Parceable to store the sidebar's current state within an instance of the class SidebarSavedState when the view is destroyed, e.g. when the activity, it belongs to, loses its focus. This method is implicitly called by the Android API. For restoring a saved state afterwards, the method onRestoreInstanceState(Parceable):void, which is also implicitly called, is overridden. Overriding these methods i.a. ensures, that the sidebar remains shown, when the orientation of the Android device changes.

The view Sidebar is designed to be the parent view for two child views: The view, which contains the main content, and the view, which contains the sidebar. These child views are implemented in the class ContentView, which contains the main content as wells as the overlay, which is shown when the sidebar is shwon, respectively the class SidebarView, which contains the sidebar and a shadow, which is rendered in front of the content. The main task of the class Sidebar is to calculate the position and size of its child views, depending on the sidebar's current state, e.g. the location of the sidebar, whether it is currently shown or whether it is currently dragged by the user. Furthermore, the different animations, which are located within the package de.mrapp.android.sidebar.animation are used to animate the ContentView and SidebarView, when the sidebar should become shown or hidden.

2 Showing or hiding the sidebar

There are multiple possibilities to show or hide the sidebar. Besides the possibility to use dedicated methods, which are provided by the class Sidebar, the sidebar can be moved by performing drag gestures on the touch screen or it can be shown or hidden by clicking the main content, the sidebar itself or the back button of the Android device. The behavior of the sidebar towards these possibilities can be freely adjusted using setter methods or XML attributes, which are discussed in the following sections. If the sidebar should be shown or hidden programatically instead, the methods showSidebar():void, hideSidebar():void or toggleSidebar():void of the class Sidebar can be used instead. Toggeling the sidebar means, that the sidebar will become shown, if it is currently hidden or vice versa. To retrieve, whether the sidebar is currently shown or not, the method isSidebarShown():boolean is provided.

These methods are only allowed to be called after the sidebar's child views are initialized, otherwise an IllegalStateException will be thrown. This may be the case, if the sidebar is instantiated programatically and one of the methods mentioned above is called before the setContentView- and setSidebarView-methods are called to inflate the sidebar's child views (see the following section for further details about these methods).

3 Observing the sidebar's state

The type SidebarListener defines an interface, which may be used by implementing classes to be notified, when a sidebar becomes shown or hidden. Therefore a class instance, which implements this interface, must be registered by using the method addSidebarListener(SidebarListener):void of the class Sidebar. This will cause the method onSidebarShown(Sidebar):void, respectively the method onSidebarHidden(Sidebar):void, to be called when the sidebar becomes shown or hidden. When no events about the sidebar's state should be received anymore, the listener can be unregistered by calling the method removeSidebarListener(SidebarListener):void of the class Sidebar afterwards.

Although all setter methods mentioned in this section can be used at runtime to adapt the attributes of an sidebar, which will cause the GUI to be refreshed, the methods should not be called while the sidebar is currently dragged or animated to avoid unexpected behavior. To ensure, that no animation is running and no dragging gesture is currently performed on the sidebar, the methods isAnimationRunning():boolean and isDragging():boolean of the class Sidebar can be used.

5 XML attributes

If the Sidebar is created by beeing specified within a XML resource file, XML attributes can be used to customize the view's behaviour and appearance. Besides the inherited attributes, which are defined by the Android API, the Sidebar does provide various custom attributes, which can be used to set the same attributes as the setter methods already mentioned above. The custom attributes are defined within the file /res/values/attrs.xml. In the following table all custom attributes will be introduced. An example on how to define a Sidebar by using a XML resource file, is available in the "Examples" section of this documentation.

For using these custom attributes within a XML resource file, the following namespace has to be added to the root node of the XML file:

xmlns:custom="http://schemas.android.com/apk/res-auto"

If the restrictions of an attribute mentioned in the table above are violated, exceptions will be thrown at runtime.

Setting the attribute maxSidebarWidth or maxSidebarWidth via XML or by calling the setter methods mentioned in the previous section, may cause unexpected behavior, if e.g. the maximum width of the sidebar is set to less than the actual offset in dp. Althought the values are checked for validity when set, it is not possible to check such constraints, because the actual width of the sidebar might be unknown at the time, these methods are called.
